# レビューフェーズ

## 0. 前提
- `[REVIEW]` モードで動いていることを明示する。
- レビュー開始前に以下を必ず実行し、結果を備考に残す。
  - `git status -sb`
  - 対象ファイルの最新内容を確認
  - `vibes/tmp/{ブランチ名}/design.md` の再読
  - 存在すれば `vibes/tmp/{ブランチ名}/review.md` の再読
- use context7

## 1. スコープ確認
- 設計で定義された 要件に合致する範囲のみレビューする。
- 要件のスコープ外の問題を見つけた場合は `OTHER TASK` として分類し、今回のレビューでは修正を要求しない。
- `vibes/tmp/` など、.gitignoreの対象のファイルはレビュー対象にしない

## 2. 指摘フォーマット
- すべてのコメントは以下の形式で記載する。
  - `[Blocking] ファイル名: 行番号 - 問題点 - 具体的な修正案`
  - `[Refactor] ...`（品質向上案）
  - `[OtherTask] ...`（別タスクで扱う提案。理由と推奨担当を記載）
- 可能な限り一次情報（公式ドキュメント）の根拠を添える。
- 明確な誤りがない場合も、テスト観点や確認方法を補足として残す。

## 3. 再レビュー時のルール
- 前回指摘した内容ごとに「継続」「撤回」「OtherTask へ移動」のいずれかを明示し、撤回理由を必ず記載する。
- レビューが完了した日時と確認に用いたコミットハッシュを備考へ追記する。

## 4. レポート
最後に以下をまとめて `vibes/tmp/{ブランチ名}/review.md` へ出力する。
- 確認日時 / コミットハッシュ / 実行したコマンド（例: `git status`, 参照したファイル一覧）。
- 指摘一覧（上記フォーマット）。
- `OtherTask` に回した内容と理由。
- やり残しや不確定事項があれば明示する。
